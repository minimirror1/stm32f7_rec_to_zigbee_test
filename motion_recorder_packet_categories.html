PC→MCU 명령 cmd = 0x04 CMD_MOTION_CTRL
사용 app 함수:
CMD_MOTION_CTRL는 모션 관련 제어 명령을 묶어서 처리하는 상위 cmd입니다. 실제 동작 종류는 payload 첫 바이트의 MotionAction 값으로 구분되며, 현재 문서는 그중 PLAY / STOP / PAUSE를 중심으로 설명합니다.

명령 개요

  • 요청 헤더는 src_id | tar_id | cmd | payload_len(LE) 순서입니다.
  • cmd = 0x04 이면 이 패킷은 모두 CMD_MOTION_CTRL 로 해석됩니다.
  • payload 첫 바이트의 action 값이 실제 동작을 결정합니다.
  • 즉 PLAY, STOP, PAUSE는 서로 다른 독립 cmd가 아니라 같은 cmd 안에서 payload[0] 값으로 구분됩니다.

MotionAction 값

  • 0x00 = MOTION_ACTION_PLAY : 모션 재생 시작
  • 0x01 = MOTION_ACTION_STOP : 현재 재생 즉시 정지
  • 0x02 = MOTION_ACTION_PAUSE : 현재 재생 일시정지
  • 0x03 = MOTION_ACTION_SEEK : 지정 시간 위치 이동
  • 이 페이지는 우선 PLAY / STOP / PAUSE를 중심으로 정리하고, SEEK는 이후 별도 상세를 추가하면 됩니다.

공통 처리 흐름

  • 통신 레이어는 먼저 cmd = CMD_MOTION_CTRL 인지 확인합니다.
  • 그 다음 payload[0] 값을 읽어 PLAY / STOP / PAUSE / SEEK 중 어떤 동작인지 분기합니다.
  • 각 action은 대응되는 app 함수로 연결되지만, 구체적인 함수 사용법은 전체 함수 탭에서 따로 설명합니다.
  • 여기서는 명령 해석과 action 분기 자체만 명령 관점으로 봅니다.

payload[0] = 0x00 : PLAY

  • action 값은 0x00 입니다.
  • 요청 예시는 00 01 04 01 00 00 처럼 payload 1바이트만 포함합니다.
  • 이 값은 "모션 재생 시작 요청"으로 해석됩니다.
  • App 함수 사용법과 구현 책임은 전체 함수 > App_MotionPlay에서 봅니다.

payload[0] = 0x01 : STOP

  • action 값은 0x01 입니다.
  • 현재 재생 중인 모션을 즉시 중단하는 용도로 사용합니다.
  • 이 값은 "현재 재생 즉시 정지"로 해석됩니다.
  • 함수 사용법은 전체 함수 > App_MotionStop에서 봅니다.

payload[0] = 0x02 : PAUSE

  • action 값은 0x02 입니다.
  • 현재 재생을 잠시 멈추고, 이후 재개 가능한 상태를 유지하는 용도입니다.
  • 이 값은 "현재 재생 일시정지"로 해석됩니다.
  • 함수 사용법은 전체 함수 > App_MotionPause에서 봅니다.

요청 바이트 예시

/* PC -> MCU : PLAY 요청 */
00 01 04 01 00 00

00 = src_id
01 = tar_id
04 = CMD_MOTION_CTRL
01 00 = payload_len (LE)
00 = payload[0] = MOTION_ACTION_PLAY
PC→MCU 명령 cmd = 0x01 CMD_PING
사용 app 함수:
CMD_PING은 장치가 살아 있고 통신 가능 상태인지 확인하는 기본 명령입니다. 현재 binary 구현 기준으로 성공 시 CMD_PONG를 payload 없이 돌려주며, 문서 상단의 장치 정보 응답 같은 추가 payload는 아직 이 cmd에 포함되지 않습니다.

명령 개요

  • 요청 헤더는 src_id | tar_id | cmd | payload_len(LE) 순서입니다.
  • cmd = 0x01 이면 CMD_PING 로 해석됩니다.
  • 현재 구현에서 요청 payload는 없습니다.
  • 브로드캐스트(tar_id = 0xFF) 또는 단일 장치 ping에 모두 사용할 수 있습니다.

현재 구현 기준 응답

  • 성공 시 응답 cmd는 CMD_PONG(0x02) 입니다.
  • 성공 응답은 STATUS_OK, payload_len = 0 입니다.
  • 즉 현재 binary 구현에서는 DI_ID, FW 버전, 상태 정보를 payload로 싣지 않습니다.
  • 실패 시에는 에러 응답으로 처리됩니다.

대표 사용 항목

  • 장치 발견 / Ping 항목이 이 cmd에 직접 대응합니다.
  • PC는 네트워크에서 응답 가능한 장치를 찾기 위해 ping을 보냅니다.
  • 현재 프로젝트에서는 장치 정보 조회보다 "응답 여부 확인" 성격이 더 강합니다.

공통 처리 흐름

  • 통신 레이어가 CMD_PING 을 확인하면 App_Ping() 을 호출합니다.
  • App_Ping() 이 true를 반환하면 CMD_PONG 성공 응답을 보냅니다.
  • false를 반환하면 실패로 간주하고 에러 응답을 보냅니다.

요청 / 응답 바이트 예시

/* PC -> MCU : broadcast ping */
00 FF 01 00 00

00 = src_id (PC)
FF = tar_id (broadcast)
01 = CMD_PING
00 00 = payload_len (LE)
/* MCU -> PC : pong success */
0A 00 02 00 00 00

0A = src_id (example device id)
00 = tar_id (PC)
02 = CMD_PONG
00 = STATUS_OK
00 00 = payload_len (LE)
PC→MCU 명령 cmd = 0x21 CMD_GET_FILE
사용 app 함수:
CMD_GET_FILE은 path 문자열 하나로 실제 대상 파일을 구분하는 범용 읽기 명령입니다. 따라서 DI_ID / MT_RP / MT_CT / MT_ST / MT_LI / MT_AT / MT_LK / 모터 Midi 파라미터 / Midi 페이지 설정 / 로그 읽기는 전용 cmd가 아니라 모두 이 상위 cmd로 읽습니다.

명령 개요

  • 요청 payload는 path_len(2 LE) + path(path_len) 입니다.
  • 응답 payload는 path_len + path + content_len + content 입니다.
  • 동일한 cmd 안에서 path 값만 바꿔 여러 설정 파일과 로그 파일을 읽습니다.

대표 읽기 대상

  • Setting/DI_ID.TXT
  • Setting/MT_RP.TXT, Setting/MT_CT.TXT, Setting/MT_ST.TXT
  • Setting/MT_LI.TXT, Setting/MT_AT.TXT, Setting/MT_LK.TXT
  • Midi/motor/{id}.ini, Midi/page/data.ini
  • Log/BOOT.TXT, Log/SENSOR.TXT

공통 처리 흐름

  • 통신 레이어가 path 길이와 payload 길이 일치를 먼저 검사합니다.
  • path를 문자열로 복원한 뒤 App_GetFile(path, out_content, max_len) 을 호출합니다.
  • 성공 시 path와 content를 함께 응답에 담고, 실패 시 ERR_FILE_NOT_FOUND 를 보냅니다.

함수 사용 위치

  • 구현 책임은 전체 함수 > App_GetFile에 모여 있습니다.
  • 읽기 전용 cmd이고, 실제 저장은 CMD_SAVE_FILE 에서 처리합니다.
  • 저장 후 비교 검증은 CMD_VERIFY_FILE 에서 처리합니다.

요청 / 응답 payload 예시

/* read request for Setting/DI_ID.TXT */
11 00 | "Setting/DI_ID.TXT"

11 00 = path_len (17, LE)
Setting/DI_ID.TXT = path
/* success response payload example */
11 00 | "Setting/DI_ID.TXT" | 02 00 | "10"

11 00 = path_len (LE)
Setting/DI_ID.TXT = path
02 00 = content_len (LE)
10 = content
PC→MCU 명령 cmd = 0x22 CMD_SAVE_FILE
사용 app 함수:
CMD_SAVE_FILE은 path와 content를 함께 보내 실제 파일 내용을 저장하는 범용 쓰기 명령입니다. 설정 파일과 Midi 파일 계열은 현재 코드 기준으로 모두 이 cmd를 통해 갱신합니다.

명령 개요

  • 요청 payload는 path_len(2 LE) + path + content_len(2 LE) + content 입니다.
  • 성공 응답 payload는 path_len + path 만 돌려줍니다.
  • 즉 응답은 "어떤 파일 저장을 수락했는가" 확인용입니다.

대표 쓰기 대상

  • Setting/DI_ID.TXT
  • Setting/MT_RP.TXT, Setting/MT_CT.TXT, Setting/MT_ST.TXT
  • Setting/MT_LI.TXT, Setting/MT_AT.TXT, Setting/MT_LK.TXT
  • Midi/motor/{id}.ini, Midi/page/data.ini
  • 로그 파일 읽기는 CMD_GET_FILE 범위이고, 이번 문서에서는 로그 저장까지 다루지 않습니다.

공통 처리 흐름

  • 통신 레이어가 path와 content 길이를 검증합니다.
  • 그 다음 App_SaveFile(path, content) 을 호출합니다.
  • 성공 시 path만 담은 성공 응답을 보내고, 실패 시 에러 응답을 보냅니다.

함수 사용 위치

  • 구현 책임은 전체 함수 > App_SaveFile에 모여 있습니다.
  • 저장 후 실제 내용이 맞는지 확인하려면 CMD_VERIFY_FILE 을 이어서 사용합니다.

요청 / 응답 payload 예시

/* write request for Setting/MT_RP.TXT = "1" */
11 00 | "Setting/MT_RP.TXT" | 01 00 | "1"

11 00 = path_len (17, LE)
Setting/MT_RP.TXT = path
01 00 = content_len (1, LE)
1 = content
/* success response payload */
11 00 | "Setting/MT_RP.TXT"

11 00 = path_len (LE)
Setting/MT_RP.TXT = path
PC→MCU 명령 cmd = 0x23 CMD_VERIFY_FILE
사용 app 함수:
CMD_VERIFY_FILE은 expected content를 같이 보내 저장된 파일 내용과 일치 여부를 확인하는 검증 명령입니다. 설정 파일과 Midi 파일을 저장한 뒤 PC가 후속 확인용으로 사용할 수 있습니다.

명령 개요

  • 요청 payload 구조는 CMD_SAVE_FILE 과 동일합니다.
  • 성공 응답 payload는 path_len + path + match(1) 입니다.
  • match = 0x01 이면 일치, 0x00 이면 불일치입니다.

대표 검증 대상

  • Setting/DI_ID.TXT
  • Setting/MT_RP.TXT, Setting/MT_CT.TXT, Setting/MT_ST.TXT
  • Setting/MT_LI.TXT, Setting/MT_AT.TXT, Setting/MT_LK.TXT
  • Midi/motor/{id}.ini, Midi/page/data.ini

공통 처리 흐름

  • 통신 레이어가 path와 expected content를 복원합니다.
  • App_VerifyFile(path, content, &match) 을 호출합니다.
  • 파일을 읽을 수 없으면 실패, 읽을 수 있으면 일치 여부를 1바이트 match 값으로 응답합니다.

함수 사용 위치

  • 구현 책임은 전체 함수 > App_VerifyFile에 모여 있습니다.
  • 저장 자체는 CMD_SAVE_FILE, 읽기는 CMD_GET_FILE 과 역할이 나뉩니다.

요청 / 응답 payload 예시

/* verify request for Setting/MT_RP.TXT = "1" */

Setting/MT_RP.TXT = path
01 00 = content_len (LE)
1 = expected content

Setting/MT_RP.TXT = path
01 = match (1 = same, 0 = different)

현재는 App_MotionPlay / Stop / Pause / Seek / Ping / GetFile / SaveFile / VerifyFile를 먼저 정리했고, 이후 다른 App 함수도 같은 형식으로 순차 추가하면 됩니다.

App 함수 App_MotionPlay(uint8_t device_id)
CMD_MOTION_CTRL 패킷에서 payload[0] = 0x00 이 들어오면 통신 레이어가 App_MotionPlay(ctx->my_id) 를 호출합니다. 이 함수는 "모션 재생 요청을 실제 장치 동작으로 연결하는 app 진입점"입니다.

언제 호출되는가

  • PC가 CMD_MOTION_CTRL + payload[0] = 0x00 을 보내면 호출됩니다.
  • 호출 주체는 통신 레이어이며, app 코드는 직접 패킷을 파싱하지 않습니다.
  • 입력 인자는 현재 장치 id 하나뿐입니다.

app 코드가 해야 할 일

  • 현재 장치가 PLAY를 받을 수 있는 상태인지 확인합니다.
  • 필요하면 반복 정책이나 현재 프로그램 상태를 읽습니다.
  • 실제 모션 플레이어 또는 상태 머신에 재생 시작을 요청합니다.
  • 재생 시작 수락이면 true, 시작 불가면 false를 반환합니다.

하면 안 되는 일

  • 패킷 바이트 파싱이나 payload_len 해석
  • 응답 프레임 생성, UART/XBee 송신
  • cmd/action 상수 정의를 app 로직 내부에 다시 구현하는 일

호출 결과를 PC는 어떻게 보는가

  • true 반환: 통신 레이어가 요청 수락으로 간주하고 성공 응답을 보냅니다.
  • false 반환: 통신 레이어가 실패로 간주하고 에러 응답을 보냅니다.
  • 즉 이 함수는 "재생 완료"가 아니라 "재생 시작 요청 수락 여부"를 알려 주는 함수입니다.

레퍼런스 코드 조각

/* 통신 레이어에서의 연결 */
case MOTION_ACTION_PLAY:
    /* 통신 레이어가 action을 해석한 결과를 app 함수로 전달 */
    ok = App_MotionPlay(ctx->my_id);
    break;
/* app 레이어 구현 예시 */
bool App_MotionPlay(uint8_t device_id)
{
    /* 다른 장치로 잘못 들어온 호출은 거부 */
    if (device_id != MY_DEVICE_ID) {
        return false;
    }

    /* 초기화나 리소스 준비가 끝났는지 확인 */
    if (!Motion_IsReady()) {
        return false;
    }

    /* 이미 재생 중이면 중복 시작을 막음 */
    if (Motion_IsPlaying()) {
        return false;
    }

    /* 설정 파일 기반 반복 정책을 현재 플레이어 상태에 반영 */
    Motion_LoadRepeatPolicyFromSettings();
    /* 현재 선택된 프로그램의 재생 시작을 요청 */
    return Motion_StartCurrentProgram();
}
App 함수 App_MotionStop(uint8_t device_id)
CMD_MOTION_CTRL 패킷에서 payload[0] = 0x01 이 들어오면 통신 레이어가 App_MotionStop(ctx->my_id) 를 호출합니다. 이 함수는 현재 재생 중인 모션을 정지 상태로 전환하는 app 진입점입니다.

언제 호출되는가

  • PC가 CMD_MOTION_CTRL + payload[0] = 0x01 을 보내면 호출됩니다.
  • 통신 레이어가 action 값을 해석한 뒤 app 함수만 호출합니다.
  • 입력 인자는 현재 장치 id 하나입니다.

app 코드가 해야 할 일

  • 현재 재생 중인 모션이 있으면 즉시 정지 요청을 내립니다.
  • 재생 타이머, 큐, 출력 상태 등 정지 시 함께 멈춰야 할 자원을 정리합니다.
  • 정지 성공이면 true, 정지 불가 또는 실패면 false를 반환합니다.

하면 안 되는 일

  • 패킷 길이 검사나 action 값 해석을 app 코드에서 다시 하지 않습니다.
  • 응답 패킷 생성이나 전송을 직접 처리하지 않습니다.
  • STOP 요청을 별도 cmd처럼 다시 정의하지 않습니다.

레퍼런스 코드 조각

case MOTION_ACTION_STOP:
    /* STOP action은 장치 ID만 넘겨 app 정지 로직으로 연결 */
    ok = App_MotionStop(ctx->my_id);
    break;
bool App_MotionStop(uint8_t device_id)
{
    /* 장치 ID 보호 */
    if (device_id != MY_DEVICE_ID) {
        return false;
    }

    /* 현재 재생 중인 모션과 관련 출력 상태를 정지 */
    Motion_Stop();
    /* stop 요청을 정상 수락했음을 통신 레이어에 알림 */
    return true;
}
App 함수 App_MotionPause(uint8_t device_id)
CMD_MOTION_CTRL 패킷에서 payload[0] = 0x02 이 들어오면 통신 레이어가 App_MotionPause(ctx->my_id) 를 호출합니다. 이 함수는 현재 재생 위치를 유지한 채 일시정지 상태로 전환하는 app 진입점입니다.

언제 호출되는가

  • PC가 CMD_MOTION_CTRL + payload[0] = 0x02 을 보내면 호출됩니다.
  • PLAY와 마찬가지로 패킷 파싱은 통신 레이어에서 끝나고 app 함수만 호출됩니다.
  • 입력 인자는 현재 장치 id 하나입니다.

app 코드가 해야 할 일

  • 현재 재생 위치와 상태를 유지한 채 출력 갱신만 멈출지, 타이머까지 멈출지 정책을 정합니다.
  • pause를 지원하지 않거나 현재 상태에서 중단이 불가하면 false를 반환할 수 있습니다.
  • pause 성공이면 이후 resume 또는 재생 재개 정책과 호환되는 상태를 남겨야 합니다.

하면 안 되는 일

  • pause를 stop처럼 처리해 재생 위치 정보를 무조건 버리지 않습니다.
  • 응답 송신이나 상태 응답 포맷을 app 함수에서 직접 다루지 않습니다.
  • 다른 action 값을 다시 판별하지 않습니다.

레퍼런스 코드 조각

case MOTION_ACTION_PAUSE:
    /* PAUSE action을 app 레이어의 일시정지 처리로 전달 */
    ok = App_MotionPause(ctx->my_id);
    break;
bool App_MotionPause(uint8_t device_id)
{
    /* 장치 ID 보호 */
    if (device_id != MY_DEVICE_ID) {
        return false;
    }

    /* 현재 위치를 유지한 채 pause 처리 */
    return Motion_Pause();
}
App 함수 App_MotionSeek(uint8_t device_id, uint32_t time_ms)
CMD_MOTION_CTRL 패킷에서 payload[0] = 0x03 이 들어오고 전체 payload_len = 5 일 때, 통신 레이어가 payload[1..4] 를 little-endian time_ms 로 해석한 뒤 App_MotionSeek(ctx->my_id, time_ms) 를 호출합니다.

언제 호출되는가

  • PC가 payload[0] = 0x03 과 seek 대상 시간 4바이트를 함께 보낼 때 호출됩니다.
  • payload_len != 5 이면 통신 레이어에서 바로 에러 응답을 보내므로 app 함수는 호출되지 않습니다.
  • 입력 인자는 현재 장치 id와 목표 시간 time_ms 입니다.

app 코드가 해야 할 일

  • time_ms 가 현재 모션 길이 범위 안에 있는지 검증합니다.
  • 내부 플레이어를 해당 시간 위치로 이동시키고, 필요하면 보간 상태나 출력 캐시를 다시 맞춥니다.
  • seek 성공이면 true, 범위 초과나 이동 실패면 false를 반환합니다.

요청 데이터 해석

  • payload[0] : MOTION_ACTION_SEEK (0x03)
  • payload[1..4] : little-endian 32비트 time_ms
  • 예: 03 F4 01 00 00 이면 time_ms = 500 ms 입니다.

레퍼런스 코드 조각

case MOTION_ACTION_SEEK:
    /* SEEK는 action 1바이트 + time_ms 4바이트가 반드시 필요 */
    if (payload_len != 5u) {
        SendErrorResponse(ctx, src_id, (uint8_t)CMD_MOTION_CTRL, ERR_INVALID_INPUT, NULL);
        return;
    }
    {
        /* payload[1..4]에서 32비트 시간을 복원 */
        uint32_t time_ms = read_u32le(payload + 1u);
        /* 해석된 목표 시간을 app seek 함수로 전달 */
        ok = App_MotionSeek(ctx->my_id, time_ms);
    }
    break;
bool App_MotionSeek(uint8_t device_id, uint32_t time_ms)
{
    /* 장치 ID 보호 */
    if (device_id != MY_DEVICE_ID) {
        return false;
    }

    /* 요청된 시간이 현재 모션 길이를 넘는지 확인 */
    if (time_ms > Motion_GetDurationMs()) {
        return false;
    }

    /* 내부 플레이어를 지정한 시간 위치로 이동 */
    return Motion_Seek(time_ms);
}
App 함수 App_Ping(void)
CMD_PING 이 들어오면 통신 레이어가 App_Ping() 을 호출합니다. 이 함수는 장치가 현재 응답 가능한 상태인지 판단하는 가장 얇은 app 진입점입니다.

언제 호출되는가

  • PC가 CMD_PING 을 보내면 호출됩니다.
  • 현재 구현에서는 입력 payload가 없으므로 함수 인자도 없습니다.
  • 대표 대응 항목은 장치 발견 / Ping 입니다.

app 코드가 해야 할 일

  • 장치가 통신 응답 가능한 상태인지 간단히 판단합니다.
  • 필요하면 초기화 완료 여부나 통신 서브시스템 상태를 확인합니다.
  • 응답 가능하면 true, 불가하면 false를 반환합니다.

PC는 어떻게 해석하는가

  • true 반환 시 통신 레이어가 CMD_PONG 성공 응답을 보냅니다.
  • false 반환 시 에러 응답으로 처리됩니다.
  • 현재 binary 구현 기준으로 PONG payload는 비어 있습니다.

하면 안 되는 일

  • PONG 프레임을 app 코드에서 직접 만들지 않습니다.
  • DI_ID, FW 버전 등 추가 payload를 여기서 임의로 붙이지 않습니다.
  • ping 용도를 모션/파일 명령과 섞지 않습니다.

레퍼런스 코드 조각

/* 통신 레이어에서의 연결 */
bool ok = App_Ping();
if (ok) {
    /* 응답 가능하면 payload 없는 CMD_PONG 성공 응답 */
    SendBinaryResponse(ctx, src_id, (uint8_t)CMD_PONG, STATUS_OK, NULL, 0u);
}
/* app 레이어 구현 예시 */
bool App_Ping(void)
{
    /* 최소 구현은 "지금 응답 가능한가"만 판단하면 충분 */
    return true;
}
App 함수 App_GetFile(const char *path, char *out_content, uint16_t max_len)
CMD_GET_FILE 요청이 들어오면 통신 레이어가 path 문자열을 복원한 뒤 App_GetFile(path, out_content, max_len) 을 호출합니다. 이 함수는 설정 파일, Midi 파일, 로그 파일 읽기의 공통 app 진입점입니다.

언제 호출되는가

  • PC가 CMD_GET_FILE 로 특정 path를 읽고 싶을 때 호출됩니다.
  • 통신 레이어가 payload에서 path를 꺼낸 뒤 app 함수만 호출합니다.
  • 입력은 path, 출력은 out_content 버퍼입니다.

대표 사용 항목

  • DI_ID 읽기/쓰기 → Setting/DI_ID.TXT
  • 반복재생 설정 / 반복횟수 / 모터 초기화 / 소프트 리밋 / 앱솔루트 / 모터 링크
  • Setting/MT_RP.TXT, MT_CT.TXT, MT_ST.TXT, MT_LI.TXT, MT_AT.TXT, MT_LK.TXT
  • 모터 Midi 파라미터 / Midi 페이지 설정 → Midi/motor/{id}.ini, Midi/page/data.ini
  • 로그 읽기 → Log/BOOT.TXT, Log/SENSOR.TXT

app 코드가 해야 할 일

  • 요청 path가 실제로 읽을 수 있는 파일인지 판단합니다.
  • 파일 내용을 max_len 을 넘지 않게 버퍼에 복사하고 null 종료를 보장합니다.
  • 읽기 성공이면 true, 파일 없음 또는 읽기 실패면 false를 반환합니다.

하면 안 되는 일

  • payload 길이 검증이나 path_len 파싱을 app 코드에서 다시 하지 않습니다.
  • 응답 프레임 작성이나 전송을 직접 처리하지 않습니다.
  • 쓰기와 검증까지 한 함수에 억지로 합치지 않습니다.

레퍼런스 코드 조각

/* 통신 레이어에서의 연결 */
/* path를 문자열로 복원한 뒤 공통 읽기 함수로 전달 */
bool ok = App_GetFile(path_buf, content_buf, APP_CONTENT_MAX_LEN);
if (!ok) {
    SendErrorResponse(ctx, src_id, (uint8_t)CMD_GET_FILE, ERR_FILE_NOT_FOUND, NULL);
}
/* app 레이어 구현 예시 */
bool App_GetFile(const char *path, char *out_content, uint16_t max_len)
{
    /* max_len 범위 안에서 텍스트 파일을 읽고 null 종료 보장 */
    return SD_ReadFile(path, out_content, max_len) == 0;
}

하면 안 되는 일

  • content_len 계산이나 payload 검증을 app 코드에서 다시 하지 않습니다.
  • 저장 성공 응답을 직접 만들지 않습니다.
  • 저장 후 일치 여부 판단까지 여기서 끝내지 않고, 필요하면 Verify 경로를 사용합니다.

레퍼런스 코드 조각

/* 통신 레이어에서의 연결 */
/* path와 content를 복원한 뒤 공통 저장 함수로 전달 */
bool ok = App_SaveFile(path_buf, content_buf);
if (!ok) {
    SendErrorResponse(ctx, src_id, (uint8_t)CMD_SAVE_FILE, ERR_FILE_NOT_FOUND, NULL);
}
/* app 레이어 구현 예시 */
bool App_SaveFile(const char *path, const char *content)
{
    /* 필요하면 app 레이어에서 path whitelist 검사 추가 */
    /* 받은 문자열을 그대로 파일에 저장 */
    return SD_WriteFile(path, content) == 0;
}
App 함수 App_VerifyFile(const char *path, const char *content, bool *out_match)
CMD_VERIFY_FILE 요청이 들어오면 통신 레이어가 path와 expected content를 복원한 뒤 App_VerifyFile(path, content, &out_match) 을 호출합니다. 이 함수는 저장된 파일 원문과 기대 문자열이 같은지 확인하는 검증용 app 진입점입니다.

언제 호출되는가

  • PC가 저장 후 같은 path/content로 검증을 요청할 때 호출됩니다.
  • 통신 레이어는 bool match 변수를 준비해 app 함수에 넘깁니다.
  • 대표 대상은 설정 파일과 Midi 파일입니다.

app 코드가 해야 할 일

  • 대상 파일을 읽을 수 있으면 실제 원문과 expected content를 비교합니다.
  • 비교를 수행할 수 있으면 true를 반환하고 결과를 out_match 에 기록합니다.
  • 파일을 읽을 수 없으면 false를 반환합니다.

PC는 어떻게 해석하는가

  • 함수가 true를 반환하면 응답 payload의 마지막 1바이트에 match 값이 실립니다.
  • match = 0x01 이면 저장 내용이 기대값과 같습니다.
  • match = 0x00 이면 읽기는 성공했지만 내용은 다릅니다.

하면 안 되는 일

  • 비교 결과를 응답 바이트로 직접 포장하지 않습니다.
  • 파일을 읽지 못했는데도 match만 false로 두고 true를 반환하지 않습니다.
  • 읽기와 저장 로직을 이 함수에 중복 구현하지 않습니다.

레퍼런스 코드 조각

/* 통신 레이어에서의 연결 */
bool match = false;
bool ok = App_VerifyFile(path_buf, content_buf, &match);
if (ok) {
    /* 성공 시 path와 match(1byte)를 응답에 실어 보냄 */
}
/* app 레이어 구현 예시 */
bool App_VerifyFile(const char *path, const char *content, bool *out_match)
{
    char buffer[APP_CONTENT_MAX_LEN];

    /* 먼저 실제 파일을 읽을 수 있어야 검증 가능 */
    if (SD_ReadFile(path, buffer, sizeof(buffer)) != 0) {
        return false;
    }

    /* 읽은 원문과 기대 문자열이 같은지 비교 */
    *out_match = (strcmp(buffer, content) == 0);
    return true;
}
